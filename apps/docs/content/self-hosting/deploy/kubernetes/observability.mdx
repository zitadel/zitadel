Zitadel runs as two independent containers — **Zitadel** (Go) and **Login** (Node.js). Both support OpenTelemetry and Prometheus, but because they are separate deployments, observability must be configured on each one independently.

| Signal | Zitadel | Login |
|--------|---------|-------|
| Traces | ✅ OTLP push | ✅ OTLP push |
| Metrics | ✅ OTLP push + Prometheus scrape | ✅ OTLP push + Prometheus scrape |
| Logs | ✅ OTLP push + stdout | ✅ OTLP push + stdout |

For the full reference on signals, env vars, sampling, propagation, TLS, and compression see the [OpenTelemetry guide](/self-hosting/manage/metrics/opentelemetry).

## OpenTelemetry

Both containers ship with OTel instrumentation built in and honour the full set of [standard OTel environment variables](https://opentelemetry.io/docs/specs/otel/configuration/sdk-environment-variables/). Set the variables on each container separately via `zitadel.env` and `login.env`:

```yaml
zitadel:
  env:
    - name: OTEL_EXPORTER_OTLP_ENDPOINT
      value: "http://otelcol.monitoring.svc.cluster.local:4317"
    - name: OTEL_EXPORTER_OTLP_PROTOCOL
      value: "grpc"

login:
  env:
    - name: OTEL_EXPORTER_OTLP_ENDPOINT
      value: "http://otelcol.monitoring.svc.cluster.local:4317"
    - name: OTEL_EXPORTER_OTLP_PROTOCOL
      value: "grpc"
```

Point the endpoint at an [OpenTelemetry Collector](https://opentelemetry.io/docs/collector/) in your cluster. The collector can then forward to Jaeger, Grafana Tempo, Loki, or any OTLP-compatible backend.

> **OpenTelemetry Operator:** If you manage your collector using the [OpenTelemetry Operator](https://github.com/open-telemetry/opentelemetry-operator) and the `OpenTelemetryCollector` CRD, you still need to set these env vars explicitly. Both containers already have OTel instrumentation built in, so the Operator's auto-instrumentation injection does not apply.

## Prometheus Metrics

> **We recommend OpenTelemetry over Prometheus scraping.** A single OTel endpoint covers traces, metrics, and logs with no additional configuration. Prometheus scraping exists for teams with an existing pull-based metrics pipeline that cannot yet adopt an OTel collector — if that's not you, configure OpenTelemetry above and skip this section.

Both containers expose metrics at `/debug/metrics`. Enable scraping on each separately.

### With Prometheus Operator (ServiceMonitor)

```yaml
metrics:
  enabled: true
  serviceMonitor:
    enabled: true
    additionalLabels:
      release: kube-prometheus-stack   # must match your Prometheus Operator's serviceMonitorSelector

login:
  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: kube-prometheus-stack
```

> **Label matching:** The `additionalLabels` must match the `serviceMonitorSelector` configured on your Prometheus Operator instance. If the labels don't match, the ServiceMonitor will be ignored and no metrics will be scraped. For `kube-prometheus-stack`, the default selector looks for `release: <helm-release-name>`.

### Without Prometheus Operator (annotation-based)

```yaml
metrics:
  enabled: true

podAnnotations:
  prometheus.io/scrape: "true"
  prometheus.io/port: "8080"
  prometheus.io/path: "/debug/metrics"

login:
  podAnnotations:
    prometheus.io/scrape: "true"
    prometheus.io/port: "8080"
    prometheus.io/path: "/debug/metrics"
```

For full Prometheus setup options including VM and Docker see the [Prometheus guide](/self-hosting/manage/metrics/prometheus).

## Logs

> **We recommend OpenTelemetry over stdout log collection.** If you have configured an OTel endpoint above, logs are already being pushed to your collector alongside traces and metrics — no further setup needed. The stdout approach below is for teams that cannot run an OTel collector and need to collect logs via a sidecar or node-level agent instead.

For setups without an OTel collector, logs can be collected from stdout by a log agent running in your cluster. Common collectors: [Fluent Bit](https://fluentbit.io/), [Vector](https://vector.dev/), [Promtail](https://grafana.com/docs/loki/latest/clients/promtail/).

To emit JSON-structured logs to stderr, set the following on the Zitadel container:

```yaml
zitadel:
  env:
    - name: ZITADEL_INSTRUMENTATION_LOG_STDERR
      value: "json"
```

The Login container emits structured JSON by default.
